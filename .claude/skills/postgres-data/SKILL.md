# Postgres + JPA + Flyway

Runtime DB: PostgreSQL. Tests: H2 (in-memory). Schema is owned by **Flyway**, not by
Hibernate `ddl-auto`. Keep `ddl-auto` at `validate` (or `none`) in real profiles so
the entities and migrations must agree.

## Flyway migrations (the golden rules)

- Migrations are **forward-only and immutable**. Never edit a migration that has been
  applied/committed — write a new one.
- Naming: `V<version>__<snake_case_description>.sql` in
  `src/main/resources/db/migration`, matching the existing style
  (`V9__add_sale_price.sql`, `V11__product_images.sql`, `V12__create_reviews_table.sql`).
  The last applied version is **V12**, so the next migration is `V13__…`. Version
  numbers strictly increase — never reuse or renumber.
- One logical change per migration. Prefer additive, backward-compatible changes so a
  rolling deploy never breaks the running app (add column nullable → backfill → add
  constraint in a later migration).
- Write raw SQL that runs on Postgres. Beware H2/Postgres dialect differences in tests;
  keep migration SQL standard where possible.
- Destructive changes (drop column/table) get their own migration and a note on why
  it's safe.

## Entities (match this repo's actual patterns)

- **PKs can be a business string.** `Product`'s `@Id` is `productNumber` (a String,
  generated by `ProductNumberGenerator` via a DB sequence, `V7`), not a `@GeneratedValue`
  surrogate. Match whatever the migration defines — don't blindly add `@GeneratedValue`.
- **Soft delete is the convention.** A `boolean deleted` column (default false); rows are
  never hard-deleted from the storefront. Repository finders bake the filter into the
  name — see the soft-delete rule below.
- **Collections: LAZY + `@BatchSize`.** `Product.images` is an `@ElementCollection`
  (`product_images` table, `@OrderColumn`) marked `FetchType.LAZY` with `@BatchSize(30)`.
  EAGER on a collection forces in-memory pagination of the parent (HHH000104) — never do
  that on a paged entity.
- Enums persisted as `@Enumerated(EnumType.STRING)` (e.g. `status`, `category`), never
  ordinal.
- Money as `BigDecimal` — never `double`. Timestamps as `Instant` with
  `@CreationTimestamp`/`@UpdateTimestamp` (`createdAt` is `updatable = false`).
- Keep a `protected` no-arg constructor for JPA plus a real constructor for app code.
- This project does **not** currently use `@Version`/optimistic locking; only add it if a
  feature genuinely has concurrent writers, and pair it with a migration column.

## Soft-delete query convention

- Storefront/active reads filter deleted rows **in the finder name or query**:
  `findByDeletedFalse`, `findByStatusAndDeletedFalse`,
  `findByProductNumberAndDeletedFalse`, `countByDeletedFalse`, etc. A soft-deleted
  product must 404 on the public detail route.
- Admin "Deleted" views use the `…DeletedTrue` counterparts.
- Any new active-data finder you add must include the `AndDeletedFalse` filter, or you'll
  leak deleted rows to the storefront.

## Queries & performance

- Avoid N+1: use `JOIN FETCH` or an entity graph when you'll access a collection.
- Paginate list endpoints (`Pageable`) — never return unbounded result sets to the API.
- Add an index (in a migration) for every column you filter/sort/join on frequently
  (e.g. foreign keys, `slug`, `created_at`). Unique constraints for natural keys.
- Push filtering to SQL; don't fetch-all-then-filter in Java.

## Testing schema changes

- H2 test profile should also run Flyway (or a compatible schema) so migrations are
  exercised. A migration that only works on Postgres but breaks H2 tests must be
  reconciled, not ignored.

## Checklist

1. Schema change has a new, higher-versioned Flyway migration (never edited an old one).
2. Entity fields/types match the migration; associations are LAZY unless justified.
3. Indexes added for new filter/join columns.
4. List endpoints paginate.
5. `mvn clean test` passes with migrations applied.
